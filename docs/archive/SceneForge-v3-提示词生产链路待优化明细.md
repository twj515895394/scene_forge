# SceneForge v3 提示词生产链路待优化明细

> 版本：v3 草案  
> 日期：2026-06-02  
> 来源：`huaqiang-watermelon-40s` 样例复盘、交接文档、会话记录、用户补充反馈与最终 4 段拼接视频观察  
> 目标：把 SceneForge 从“积极自动产出”调整为“可控、可确认、面向提示词生产的导演流水线”。

---

## 1. 背景

本轮围绕 `projects/huaqiang-watermelon-40s` 样例暴露出一批关键问题：

1. 阶段执行容易跳步，例如从设计阶段直接推进到表演导演，缺少剧本改编阶段的显式确认。
2. 用户只是提出纠错或讨论意见时，Agent 容易直接落盘修改并推进状态。
3. 角色、场景、道具、故事板、视频分段提示词都仍偏 v1，可用但不够贴合真实视频生成工作流。
4. 4 段 10 秒视频拼接后，单段内部基本成立，但段落之间衔接不够自然。
5. 重点桥段没有被故事板和视频提示词足够强化，例如第二次“这瓜保熟吗？”、“翻秤盘露吸铁石”、“萨日朗！！！”。
6. 当前项目边界表述不够准确，容易让 Agent 声称自己已经生成图片或视频。

v3 的核心目标不是增加更多自动化，而是把每一步变得更可控、更适合用户手动使用外部图像/视频平台进行生成。

---

## 2. 项目边界修正：SceneForge 只生产提示词，不生成图片和视频

### 2.1 正确边界

SceneForge 当前项目只负责输出：

- 角色设定图提示词；
- 场景设定图提示词；
- 道具设定图提示词；
- 故事板图提示词；
- 视频分段生成提示词；
- 音频、音乐、拟音、混音提示词；
- 发布文案、字幕、标题等文本类产物。

SceneForge 本身不负责：

- 生成图片；
- 生成视频；
- 生成音频；
- 调用绘图引擎；
- 调用视频模型；
- 将生成结果嵌入仓库。

### 2.2 允许在提示词中写“参考图”

“项目不生成图片/视频”不等于提示词不能引用外部生成的参考图。

正确口径是：

- 用户可以先根据 SceneForge 输出的角色设定提示词，在外部平台手动生成角色设定图。
- 用户可以根据场景/道具提示词，在外部平台手动生成场景与道具参考图。
- 故事板图提示词可以明确写：参考已生成的角色设定图、场景图、道具图，生成故事板画面。
- 视频分段提示词可以明确写：参考角色设定图、场景道具图、故事板图，生成对应 10 秒或 15 秒视频片段。

### 2.3 禁止表述

Agent 不得使用以下表述：

```text
我已经生成了角色图。
我已经生成了故事板图。
我已经生成了视频。
我后台调用绘图引擎运行了提示词。
我已经把生成图片嵌入文档。
```

### 2.4 推荐表述

应统一改成：

```text
已生成用于制作角色设定图的提示词。
已生成用于制作故事板图的提示词。
已生成用于基于角色图、场景道具图、故事板图生成视频的分段提示词。
这些图片和视频需要用户在外部平台手动生成。
```

## 3. 当前执行链路问题

## 3.1 阶段跳转失控

交接文档中的目标链路是：

```text
scene-topic-gate
-> scene-reference-decider
-> scene-asset-checker
-> scene-design-builder
-> scene-script-adapter
-> scene-performance-director
-> scene-storyboard-director
-> scene-audio-director
-> scene-video-prompt-builder
-> scene-publish-review
```

但 `huaqiang-watermelon-40s` 样例中，用户确认继续后，Agent 直接报告完成 `scene-performance-director`，中间的 `scene-script-adapter` 没有充分展示、确认和沉淀。

这会导致后续所有表演、分镜、音频、视频提示词都缺少经过用户确认的剧本依据。

### v3 要求

1. `/scene-forge` 总控必须只执行 `PROJECT_BOARD.md` 中的 `next_stage`。
2. 用户说“继续”只能解释为执行当前 `next_stage`，不能一口气连跑多个阶段。
3. 若用户没有明确授权，Agent 不得跳过阶段。
4. 每次执行前必须展示：
   - 当前 `project_status`；
   - 当前 `next_stage`；
   - 本次将产生或修改的文件；
   - 本次是否会写入仓库。

---

## 3.2 用户纠错被当成自动执行授权

在会话中，用户指出“摊主有两个人”“电子秤吸铁石在秤盘反面”“核心道具可否放在一张 16:9 图里”等问题时，Agent 多次直接修改文件、更新 Prompt、推进 PROJECT_BOARD。

用户的纠错和讨论，不应默认等于授权落盘。

### v3 要求

默认行为：

```text
用户提出问题 / 纠错 / 建议
-> Agent 先复述理解
-> 给出可选修改方案
-> 等用户确认是否落盘
-> 再修改文件
```

除非用户明确说：

```text
更新到文件
直接改
写入仓库
按这个方案执行
```

否则不得自动落盘。

---

## 3.3 缺少阶段确认闸门

当前链路更像“连续自动创作”，而不是“导演流水线”。

v3 应该增加 `pending_review` 思路：每个关键阶段先输出方案草案或候选项，由用户确认后再进入正式提示词产出。

建议状态：

```text
duration_strategy_pending_review
asset_plan_pending_review
design_direction_pending_review
script_plan_pending_review
script_ready_pending_review
character_prompt_plan_pending_review
scene_prop_prompt_plan_pending_review
storyboard_plan_pending_review
video_prompt_plan_pending_review
```

确认后再进入：

```text
design_ready
script_ready
performance_ready
storyboard_ready
audio_ready
video_prompts_ready
```

## 4. 前置时长与分段策略必须提前确认

## 4.1 问题

在本次样例中，总时长 40 秒是用户提出的，但单段生成时长默认采用 10 秒，没有系统性询问用户：

- 是否默认 10 秒；
- 是否使用 15 秒；
- 是否采用混合分段；
- 所用视频平台是否支持稳定 15 秒生成。

这一步非常关键，因为它会直接影响：

- 剧本节奏；
- Story Beat 数量；
- 台词保留密度；
- 镜头数量；
- 桥接分镜数量；
- 每段视频参考图配置；
- 每段视频首尾动作设计。

## 4.2 v3 要求

在 `scene-script-adapter` 之前，必须增加“时长与分段策略确认”。

推荐询问用户：

```text
你希望单段视频生成时长采用哪种？

A. 默认 10 秒一段
- 稳定性高
- 兼容多数视频模型
- 分段更多，衔接点更多

B. 15 秒一段
- 单段表演更完整
- 台词和停顿空间更充足
- 对模型稳定性要求更高

C. 混合分段
- 根据剧情节奏决定，例如 10+10+10+10 或 15+10+15
- 更贴合剧情，但规划复杂度更高
```

## 4.3 PROJECT_BOARD 建议字段

```yaml
target_total_duration_seconds: 40
segment_duration_seconds: 10
segment_strategy:
  mode: fixed_10s
  segments:
    - segment_id: SEG01
      duration_seconds: 10
    - segment_id: SEG02
      duration_seconds: 10
    - segment_id: SEG03
      duration_seconds: 10
    - segment_id: SEG04
      duration_seconds: 10
duration_confirmed_by_user: true
duration_confirmation_notes: "用户确认采用 40 秒总长，单段 10 秒，共 4 段。"
```

混合分段示例：

```yaml
target_total_duration_seconds: 40
segment_duration_seconds: mixed
segment_strategy:
  mode: mixed
  segments:
    - segment_id: SEG01
      duration_seconds: 15
    - segment_id: SEG02
      duration_seconds: 10
    - segment_id: SEG03
      duration_seconds: 15
duration_confirmed_by_user: true
```

## 5. 剧本改编阶段优化

## 5.1 问题

本次样例中，剧本改编没有充分给用户展示：

- 40 秒版本准备保留哪些原台词；
- 哪些台词删减或压缩；
- 哪些动作去害化；
- 哪些桥段是重点；
- 每段 10 秒承担什么剧情任务；
- 第二次“这瓜保熟吗？”如何设计气氛；
- “萨日朗！！！”是否作为结尾记忆点。

## 5.2 v3 要求

`scene-script-adapter` 不应直接写完整剧本，而应先输出“剧本改编方案预览”。

示例：

```text
方案 A：高度还原原台词，40 秒压缩版
方案 B：保留经典台词，强化动画喜剧节奏
方案 C：短视频爽感版，更强调爆点和转场
```

每个方案必须列出：

- 总时长；
- 分段策略；
- Story Beat 数量；
- 每个 Beat 的核心台词；
- 必须原样保留的台词；
- 可压缩或改写的台词；
- 去害化动作方案；
- Hero Moment；
- 风险点。

用户确认后，才生成正式剧本提示词或剧本文档。

## 6. 角色设定提示词优化

## 6.1 当前问题

当前角色 Prompt 基本是：

- 中间主角色全身；
- 左侧侧面和背面；
- 右上 3 个表情；
- 右下 2 个姿态。

这对于基础角色图是够的，但不适合后续故事板和视频多图参考。

主要问题：

1. 表情矩阵太少。
2. 表情没有从剧本桥段提取。
3. 动作姿态太少。
4. 视角不完整。
5. 缺少角色与核心道具的交互关系。
6. 缺少角色之间体型比例对照。
7. 缺少和剧情节点绑定的表演状态。
8. 缺少物理逻辑与安全边界。
9. 角色板缺少编号、板块标题和说明文字。

## 6.2 v3 角色设定图标准模板

默认角色设定图 Prompt 应包含：

```text
1. 角色主视图
- 正面全身
- 清晰站姿
- 服装材质锚点

2. 多视角转面
- 正面
- 3/4 视角
- 侧面
- 背面

3. 表情矩阵
- 6 到 9 个表情
- 必须从剧本和 Story Beat 中提取

4. 身体动作矩阵
- 4 到 6 个关键姿态
- 必须从剧情动作中提取

5. 道具交互姿态
- 角色与摩托、头盔、电子秤、西瓜、刀、小板凳等的正确交互方式

6. 体型比例与剪影
- 与其他角色的比例关系
- 三人同框比例参考

7. 物理与安全边界
- 刀不指向人
- 刀只作用于西瓜
- 爆汁是西瓜汁，不是血
- 摔倒是卡通物理，不是真实伤害

8. 板块编号和说明文字
- 便于用户理解、归档和上传参考图
```

## 6.3 角色板默认允许有文字

v3 默认不再要求“无文字干净版”。

理由：

- 当前角色图主要作为制作资料板和多图参考输入。
- 用户需要通过编号、板块标题、说明文字理解每个区域。
- 这些图不是作为最终视频的首尾帧直接进入画面。
- 文字对最终视频生成影响较小，且可帮助多图参考模型理解结构。

仅当用户明确需要纯视觉图时，才额外输出“无文字版 Prompt”。

## 6.4 针对《华强买瓜》的角色表情矩阵建议

### 华强

```text
- 平静问价
- 冷静调侃“金子做的”
- 第一次问“保熟吗”的试探
- 第二次问“这瓜保熟吗？”的压迫凝视
- 发现秤有问题的低头审视
- 翻秤盘后的冷静确认
- 高潮后重新恢复平静
- 骑车离场时的淡定背影
```

### 摊主老板

```text
- 漫不经心剔牙
- 被调侃价格后的暴躁
- 听到第一次“保熟吗”的不耐烦
- 听到第二次“这瓜保熟吗？”后意识到对方找茬
- 强装凶狠
- 秤被揭穿后的瞳孔收缩
- 心虚卡壳
- 恼羞成怒
- 被西瓜汁冲击后的狼狈委屈
```

### 小弟

```text

- 狗仗人势附和
- 看戏嗑瓜子
- 发现不对劲后的缩脖子
- 翻秤盘时的震惊
- 爆汁瞬间的四肢乱蹬
- 破音大喊“萨日朗！！！”
- 一只鞋飞出后的狼狈逃跑
```

## 7. 场景与道具提示词优化

## 7.1 当前问题

当前已有“核心道具与载具设定合集板”，把摩托、电子秤、水果刀、西瓜放入一张 16:9 图中，这是正确方向。

但仍有不足：

1. 场景图和道具合集图分离。
2. 核心道具合集未覆盖完整场景空间关系。
3. 道具状态没有系统状态机。
4. 小板凳、瓜子、牙签等次级道具未纳入统一参考。
5. 没有明确适配“视频模型最多上传 5 张参考图”的限制。

## 7.2 v3 新增：全场景资产总参考图 Prompt

新增一类 Prompt：

```text
全场景资产总参考图 / Master Scene-Prop Reference Board
```

目标：用一张 16:9 图同时表达：

- 街角水果摊主场景；
- 角色站位空间关系；
- 核心道具位置；
- 道具关键状态；
- 物理交互关系；
- 后续视频多图参考的总锚点。

建议布局：

```text
左侧：街角水果摊主场景
- 灰色老墙
- 红蓝遮阳棚
- 木箱西瓜堆
- 桌面和电子秤位置

中部：角色站位区
- 华强靠近西瓜摊
- 摊主在摊位内侧
- 小弟在侧后方小板凳区域
- 摩托停在摊位外侧

右侧：核心道具状态矩阵
- 摩托车：入场 / 停靠 / 离场
- 电子秤：正常称重 / 翻开秤盘露磁铁 / 受损变形
- 西瓜：完整 / 敲击 / 上秤 / 切开 / 爆汁
- 水果刀：闲置 / 拿起 / 圆头安全刀切西瓜
- 小板凳、瓜子、牙签等辅助道具

下方：比例与物理说明
- 刀只作用于西瓜
- 磁铁只在秤盘翻开后露出
- 爆裂物是西瓜汁和果肉，不是血
```

## 7.3 道具状态机

每个核心道具都应有状态机，供故事板和视频提示词引用。

### 电子秤

```text
Scale_State_A：正常称重，秤盘平放，显示读数
Scale_State_B：华强观察秤盘，准备翻开
Scale_State_C：秤盘被翻开，反面吸附红色圆形磁铁
Scale_State_D：摊主心虚，秤盘仍处于暴露状态
Scale_State_E：高潮后滑稽受损状态
```

### 西瓜

```text
Melon_State_A：完整圆瓜
Melon_State_B：被手敲击，藤蔓弹动
Melon_State_C：放上电子秤
Melon_State_D：被安全圆头刀切开
Melon_State_E：红色西瓜汁、果肉、瓜籽爆开
```

### 水果刀

```text
Knife_State_A：放在摊位边缘
Knife_State_B：摊主手摸向刀
Knife_State_C：刀进入画面，但不指向人
Knife_State_D：刀只作用于西瓜
Knife_State_E：高潮后离开人物身体范围
```

### 摩托车

```text
Motor_State_A：入场疾驰
Motor_State_B：Q 弹刹停
Motor_State_C：停靠在摊位外侧
Motor_State_D：华强重新上车
Motor_State_E：离场
```

## 8. 故事板优化

## 8.1 当前问题

当前 12 个主分镜能覆盖主要剧情，但缺少“视频拼接设计”。

问题包括：

1. SEG01 到 SEG02 缺少从对峙转向敲瓜的桥接。
2. SEG02 到 SEG03 缺少从保熟对话转向上秤的桥接。
3. SEG03 到 SEG04 缺少从翻秤盘揭穿转向高潮动作的桥接。
4. 重点看点没有被标记为 Hero Shot。
5. “萨日朗！！！”没有成为明确主镜头。
6. 第二次“这瓜保熟吗？”的压迫氛围不足。

## 8.2 桥接帧改为桥接分镜

v3 不新增复杂概念，直接在故事板中增加桥接分镜。

建议从 12 个分镜升级为 15 或 16 个分镜。

示例：

```text
SEG01:
SH001 登场刹停
SH002 脱盔凝视
SH003 价格调侃对峙
SH004 桥接：华强视线落到西瓜，手伸向西瓜

SEG02:
SH005 敲瓜特写
SH006 第一次/第二次“这瓜保熟吗？”
SH007 摊主意识到对方找茬，压迫感特写
SH008 桥接：摊主恼怒把瓜放上秤

SEG03:
SH009 上秤读数
SH010 华强指出秤有问题
SH011 翻开秤盘露出吸铁石
SH012 桥接：摊主心虚转恼怒，手摸向刀或西瓜

SEG04:
SH013 安全化刀切西瓜
SH014 西瓜爆汁冲击摊主
SH015 小弟“萨日朗！！！”大喊特写
SH016 华强骑车离场
```

## 8.3 新增 shot_type

每个分镜必须标记类型：

```yaml
shot_type: setup_shot | action_shot | bridge_shot | hero_shot
```

说明：

```text
setup_shot：建立空间、人物、情境
action_shot：推动动作或剧情
bridge_shot：负责 Segment 之间的衔接
hero_shot：全片看点、传播点、情绪爆点
```

## 8.4 Hero Shot 机制

故事板必须明确哪些桥段是重点看点。

《华强买瓜》40 秒版至少包括：

### Hero Shot 1：第二次“这瓜保熟吗？”

```text
画面重点：摊主脸部特写
```

## 9. 视频分段提示词优化

## 9.1 当前问题

4 段视频拼接后暴露的问题：

1. SEG01 到 SEG02：从摊主发怒突然切到西瓜特写，缺少华强转向西瓜、伸手敲瓜的桥接。
2. SEG02 到 SEG03：从摊主指人突然进入上秤，缺少把西瓜放上秤的动作桥。
3. SEG03 到 SEG04：从翻秤盘露磁铁突然进入刀和爆汁，缺少摊主恼羞成怒摸刀或动作升级桥。
4. SEG04 爆汁高潮红色液体太多时，容易被误解成血腥，需要更强西瓜汁语义。
5. 小弟“萨日朗！！！”没有成为主看点。
6. 每段视频内部有画面，但段落之间没有明确的 `continuity_out` 和 `continuity_in`。

## 9.2 多图参考是默认主方案

v3 默认采用多图参考视频生成逻辑，而不是首尾帧延伸法。

推荐每段视频参考图配置：

```text
Ref 1：角色设定总参考图
Ref 2：场景道具总参考图
Ref 3：本段起始分镜图
Ref 4：本段中段分镜图
Ref 5：本段结束/桥接分镜图
```

如平台支持更多参考图，可增加 Hero Shot 或道具状态图。

## 9.3 视频分段 Prompt 必须新增字段

```yaml
continuity_in_visual: "本段开始时承接上一段结尾的角色站位、视线方向、道具状态。"
continuity_out_visual: "本段最后 1 秒停在可被下一段承接的动作或构图上。"
prop_state_in: []
prop_state_out: []
character_position_in: []
character_position_out: []
hero_moment: "本段最重要的看点。"
reference_image_plan:
  ref_1: "角色设定总参考图"
  ref_2: "场景道具总参考图"
  ref_3: "本段起始分镜"
  ref_4: "本段中段分镜"
  ref_5: "本段结束/桥接分镜"
no_jump_constraints:
  - "不要突然改变角色站位"
  - "不要突然让道具出现在手中"
  - "必须表现拿起、放下、转身、放上秤等物理动作"
```

## 9.4 视频提示词前缀建议

```text
参考上传的角色设定图、场景道具总参考图，并根据故事板中 SHxxx-SHxxx 的画面内容生成本段视频。保持角色外形、服装材质、场景结构、道具状态和上一段结尾动作的连续性。
```

## 9.5 每段必须有结尾交接设计

示例：

```text
SEG01 continuity_out:
华强的视线从摊主转向西瓜堆，右手缓慢伸向最近的一颗圆西瓜，画面停在手即将触碰瓜皮的瞬间。

SEG02 continuity_out:
摊主恼怒地抓起被敲过的西瓜，重重放上电子秤，液晶屏刚开始跳动，画面停在数字即将稳定前。

SEG03 continuity_out:
红色磁铁暴露后，摊主表情从心虚卡壳转为恼羞成怒，右手向摊位边缘的圆头西瓜刀移动。

SEG04 continuity_out:
小弟破音大喊“萨日朗！！！”后，华强重新戴上头盔，摩托车排气管喷出白烟圈，驶离画面。
```

## 10. 音频与声音提示词优化

虽然本次重点是视觉提示词与视频衔接，但声音也需要和 Hero Shot 绑定。

建议：

1. 第二次“这瓜保熟吗？”前后加入短暂降噪和压迫静默。
2. 翻秤盘露吸铁石时加入金属“叮”和 0.3 秒静默。
3. 西瓜爆汁强调“水球爆开、果肉拍脸、瓜籽弹跳”，避免血腥音效。
4. “萨日朗！！！”必须作为结尾声音 Hero Moment，音量、破音、空间回声都要明确。

## 11. 输出格式优化

所有最终可复制提示词必须：

5. 明确说明用户需要手动上传哪些参考图。
6. 明确说明该 Prompt 是用于生成什么类型的图或视频。

## 14. 结论

v3 的核心不是让 SceneForge 更自动，而是让它更像一个可控的导演制片流程。

关键变化：

```text
从：Agent 自动理解、自动生成、自动推进
到：Agent 提供方案、用户确认、再生成提示词
```

对 `huaqiang-watermelon-40s` 这类项目，v3 最重要的改造是：

1. 前置确认时长与分段策略。
2. 剧本阶段明确保留台词、Hero Moment 和去害化方案。
3. 角色提示词从“普通设定板”升级为“剧本驱动的角色制作资料板”。
4. 场景道具从“单体图”升级为“全场景资产总参考图 + 道具状态机”。
5. 故事板从“关键画面集合”升级为“主分镜 + 桥接分镜 + Hero Shot”。
6. 视频提示词从“每段内部描述”升级为“多图参考 + 开头承接 + 结尾交接 + 状态连续”。

只有这样，SceneForge 输出的提示词才能真正服务于用户在外部平台手动生成角色图、场景图、故事板图和多段视频的完整工作流。
