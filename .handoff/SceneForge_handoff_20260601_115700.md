# SceneForge Handoff - 20260601_115700

## 一、项目名称

- 英文名：SceneForge
- 中文名：名场面工坊
- 定位：经典影视 / 热点片段的 3D 动画电影化再创作生产系统。

## 二、项目方向

用户希望把经典电视剧、经典电影、热点影视片段，通过一套可复用 SOP 转成高品质 3D 动画电影化内容。

核心路径是：

1. 先选题材/经典片段。
2. 做角色风格设定，使用“高品质 3D 动画电影质感”，服装和造型做动画化改造。
3. 在角色设定图/锁定卡基础上，对经典片段剧本改编或扩写。
4. 分镜必须具有 3D 动画电影的拍摄特点和味道，不能死套原版演绎和拍摄手法。
5. 根据分镜生成故事板，可以生成多个故事板版本。
6. 根据故事板进行视频分段生成。

## 三、已达成共识的核心决策

### 1. 题材路线

- 第一阶段：国产经典电视剧名场面。
- 第二阶段：经典港片 / 国产电影名场面。
- 不限定具体类型，因为项目需要支持热点快反，经常可能临时做一个话题片段。

### 2. 选题闸门

所有片段进入制作前，先做选题评分。

最低制作门槛：60 分。

- 80 分以上：重点制作池。
- 60～79 分：快速制作池。
- 40～59 分：备选观察池。
- 低于 40 分：放弃池。

评分权重：

| 维度 | 权重 |
|---|---:|
| 热点价值 | 25 |
| 动画化适配度 | 20 |
| 改编空间 | 15 |
| 经典/认知锚点 | 15 |
| 风险可控性 | 10 |
| 制作成本可控性 | 10 |
| 平台传播潜力 | 5 |

### 3. 快速制作与重点制作

- 80 分以上重点制作：完整角色设定图、多版故事板、精修视频。
- 60～79 分快速制作：可以跳过完整角色设定图，但必须有轻量角色锁定卡。

### 4. 剧本改编档位

不再只允许少量保留台词，改成四档：

1. 原样保留型。
2. 大部分保留型。
3. 部分改写型。
4. 重构扩写型。

像《西游记》这类公版原著/神话母题，可进入原样保留型或大部分保留型；但如果参考的是某个具体影视版本，仍需区分该版本的具体改编表达。

### 5. 参考对象规则

经典片段进入 SOP 前，必须先明确：

- 本次参考的是原著/母题；
- 还是具体影视版本。

两者都支持，但不能混着默认处理。

### 6. 角色外观参考强度

不做过度保守的一刀切，改成三档：

1. 高锚点参考型：适合《西游记》等大众集体记忆非常强、且更接近原著母题传播的内容。
2. 中度参考型：适合一般经典影视角色，保留角色气质、身份感、服装轮廓和标志性元素。
3. 低参考 / 原创转译型：适合热点强 IP、高风险内容、明星脸辨识度过高的内容。

### 7. 资产复用判断

- 角色资产复用判断：强制前置。
- 场景资产复用判断：强制前置。
- 道具资产判断：轻量备选项。

道具不默认做重型资产判断，只有满足角色识别、剧情动作、系列复用任意一项时，才进入判断。

### 8. 资产版本机制

角色、场景、核心系列道具入库时必须有版本号。

示例：

- 角色_孙悟空_v1.0_经典锚点版.md
- 场景_同福客栈大堂_v1.0_白天暖光版.md
- 道具_金箍棒_v1.0_经典识别版.md

### 9. 项目架构

最终采用四层结构：

1. Skill 层：管理 SOP、流程、规则、模板。
2. assets 层：长期复用角色、场景、核心道具。
3. projects 层：每次具体片段创作的工作区。
4. outputs 层：可直接使用的最终产物。

### 10. Skill 放置位置

Skill 应放在：

```text
.agents/skills/
```

### 11. Skill 架构

采用：1 个总控 Skill + 多个 SOP 子 Skill。

总控 Skill：

- scene-forge：只做路由、编排、状态合并，不直接产出具体内容。

子 Skill：

- scene-topic-gate：选题评分。
- scene-reference-decider：参考对象判断。
- scene-asset-checker：资产复用判断。
- scene-design-builder：角色与场景设定。
- scene-script-adapter：剧本改编。
- scene-storyboard-director：分镜设计。
- scene-video-prompt-builder：故事板与视频提示词。
- scene-publish-review：发布与复盘。

### 12. Project Blackboard 黑板机制

取消“每个 Skill 都产出完整重复文档”的设计。

改为：

- 每个项目一个 `PROJECT_BOARD.md`。
- `PROJECT_BOARD.md` 是项目唯一状态源。
- 子 Skill 只输出黑板补丁块。
- 总控 Skill 负责合并补丁和更新状态。

### 13. details 与 outputs 边界

`details/`：可选长内容区，不保存 Agent 思考过程，不保存废稿。

只保存：

- 完整剧本。
- 完整分镜表。
- 复杂参考对象分析。
- 复杂风险说明。
- 角色/场景长设定草案。

`outputs/`：只保存可直接使用的最终产物。

包括：

- 故事板提示词。
- 视频分段提示词。
- 发布文案。
- 最终图片。
- 最终视频。
- 字幕。
- 配音文案。

视频提示词暂不按模型分类。

### 下一批建议问题

1. 子 Skill 的黑板补丁块格式是否需要固定为 HTML 注释包围。
2. 项目状态机字段是否最终确认：draft、topic_scored、reference_decided、assets_checked、design_ready、script_ready、storyboard_ready、video_prompts_ready、published、reviewed、archived。
3. `PROJECT_BOARD.md` 的 YAML 字段是否需要精简或补充。
4. 角色资产卡模板字段是否确认。
5. 场景资产卡模板字段是否确认。
6. 核心道具资产卡模板字段是否确认。
7. 快速制作池与重点制作池是否需要不同模板。
8. 剧本改编输出是否要区分“短视频版”和“完整分镜前置版”。
9. 分镜表字段是否要固定。
10. 故事板提示词是否要支持多版本：忠于情绪版、动画强化版、短视频爆点版。
11. 视频分段提示词是否要固定输出“总提示词 + 分段提示词”。
12. 发布文案是否要按平台区分：小红书、抖音、B站、X。
13. 风险规则是否需要单独加入“禁止项清单”。
14. 是否需要自动化项目初始化脚本。
15. 是否需要示例项目，比如《西游记》三打白骨精。

## 六、重要注意事项

1. 正式提示词中不建议直接使用“皮克斯官方风格”“Pixar 同款”等表达，应使用“高品质 3D 动画电影质感”等通用表述。
2. 项目目标不是原片复刻，也不是明星脸动画化，而是“经典情绪核 + 动画电影化重构”。
3. 但对于《西游记》等大众集体记忆强、且接近公版母题的内容，不应过度保守，应允许高锚点参考。
4. 角色和场景是最重要的复用资产，道具只做轻量备选。
5. `PROJECT_BOARD.md` 是唯一状态源，不能让 details 或 outputs 维护独立状态。
6. `details/` 不是过程垃圾桶，不保存 Agent 内部思考。
7. `outputs/` 只放最终可用产物。
