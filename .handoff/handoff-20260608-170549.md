## 仍然存在的核心问题

### 1. 规则已经越来越多，但“可执行性”仍不够硬

虽然这轮已经把很多 gate 写成了更硬的协议，但目前系统仍然以大量 prose 规则驱动：

- `SKILL.md`
- `references/output-contract.md`
- `board-protocol.md`

这三层经常会出现以下风险：

- 同一规则被重复描述
- 口径轻微漂移
- 执行层“看起来读了协议”，但仍能绕开真正意图

也就是说，当前问题已经不只是“缺一条规则”，而是**规则架构本身太依赖长文本解释**。

### 2. 当前系统缺少更机器可校验的阶段 schema / validator

本轮虽然已经把很多 review 条件写成“缺字段即失败”，但它们仍停留在协议文本层，而不是独立的、统一可执行的校验器。

高概率下一步应该考虑：

- 是否为 `storyboard` 和 `video_prompts` 阶段建立更正式的 machine-readable schema
- 是否增加阶段级 validator / linter
- 是否把 `quality_check` 从“路径 + 文字约束”提升成结构化、可自动比对的结果

### 3. 中间稿与主交付的分层仍然是脆弱点

这轮已经反复暴露：

- shotlist / director notes / parameter tables / compiled prompt / explanatory draft

很容易被执行层误当成：

- 正式主交付

说明当前系统在“中间稿 vs 主交付”的对象模型上还不够清晰。后续架构设计建议重点看：

- 是否需要统一的 artifact taxonomy
- 是否需要对每个 stage 定义：
  - `preview_artifacts`
  - `draft_artifacts`
  - `review_artifacts`
  - `final_delivery_artifacts`
- 并把 `primary/default_read/completed` 只绑定到 final delivery

## 为什么下一次会话值得重新审视架构设计

这次不是单个 bug 了，而是已经暴露出一组系统性征兆：

1. 规则大量存在，但仍反复被绕过
2. 规则多分散在多个文档层次
3. 主交付和中间稿边界容易模糊
4. 每次修复都在继续加 prose gate，复杂度在上升
5. 用户体验上最痛的地方，已经不是“少一个字段”，而是：
   - 为什么 agent 还会生成错体裁
   - 为什么 review 没真正拦住
   - 为什么一个阶段的确认会影响另一个阶段

因此，下一次会话不建议只继续“加更多协议条文”，而应考虑更高层次的架构问题：

- SceneForge 是否需要从“文档驱动状态机”升级为“文档 + 可执行校验器”的双层架构
- 是否需要统一 stage artifact model
- 是否需要把 preview / draft / review / final 四层状态做成标准模板
- 是否需要更集中化的 rule engine，而不是分散在每个 skill 文本里
